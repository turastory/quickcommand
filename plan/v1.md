# quickcommand v1 계획

## 요약
- `quickcommand`라는 Rust 단일 바이너리 CLI를 만들고 실행 파일 이름은 `qc`로 둡니다.
- 기본 동작은 `자연어 입력 -> 쉘 명령 생성 -> 터미널에 출력 -> 클립보드 복사`입니다.
- 실제 실행은 기본값이 아니며, `--execute`를 붙였을 때만 수행합니다.
- 아키텍처는 provider 확장 가능하게 설계하되, v1 구현체는 `Ollama`만 제공합니다.
- `pls`의 다중 턴 에이전트 루프는 가져오지 않고, v1은 “단일 쉘 스니펫 생성기”로 단순화합니다.

## 핵심 구현
- CLI 인터페이스
  - `qc "<task>"`: 기본 복사 모드. 생성된 명령을 출력하고 클립보드에 복사한 뒤 종료.
  - `qc --execute "<task>"`: 생성된 명령을 출력한 뒤 확인 정책에 따라 실행.
  - `qc init`: 초기 설정 마법사.
  - `qc config show`: 현재 유효 설정 출력.
  - `qc --model <name>`, `qc --host <url>`: 1회성 Ollama override.
  - `qc --confirm <all|destructive|none>`: 실행 모드에서만 적용.
- 설정
  - 설정 파일 경로는 `~/.config/quickcommand/config.toml`.
  - 기본값은 `provider="ollama"`, `mode="copy"`, `confirm_mode="destructive"`, `ollama_host="http://localhost:11434"`.
  - 기본 모델은 `qwen3.5:9b`를 우선 사용하고, 없으면 `ollama list` 결과 중 첫 번째 설치 모델로 fallback, 그것도 없으면 `init`에서 직접 입력받습니다.
  - 우선순위는 `CLI > 환경변수 > config 파일 > 기본값`.
  - 환경변수는 `QC_PROVIDER`, `QC_MODE`, `QC_CONFIRM`, `QC_OLLAMA_HOST`, `QC_OLLAMA_MODEL`을 지원하고, 호스트는 `OLLAMA_HOST`도 fallback으로 읽습니다.
- LLM 호출 방식
  - Ollama의 `/api/chat`를 `stream=false`로 호출합니다.
  - 툴 호출 루프는 쓰지 않고, JSON schema 기반 structured output만 사용합니다.
  - 모델 응답 스키마는 둘 중 하나만 허용합니다.
  - `command` 응답: `summary`, `command`.
  - `clarification` 응답: `question`, `options`, `recommended_index`.
  - clarification은 최대 2회까지만 허용하고, 그 안에 확정되지 않으면 실패로 종료합니다.
  - 프롬프트는 “최소 충분 명령”, “과도한 파괴 명령 회피”, “사용자 언어에 맞춰 요약”, “붙여넣기 가능한 완전한 쉘 스니펫 반환”을 강제합니다.
- 복사/실행 정책
  - 복사 모드에서는 항상 생성된 `command` 원문만 클립보드에 복사합니다.
  - 실행 모드에서는 기본적으로 복사하지 않고, 출력 후 확인 절차를 거쳐 실행만 합니다.
  - 실행은 사용자의 `$SHELL -lc`로 수행하고, 없으면 `/bin/sh -lc`를 사용합니다.
  - stdout/stderr는 실시간 또는 준실시간으로 그대로 보여주고 종료 코드를 반영해 종료합니다.
- 안전장치
  - 로컬 destructive matcher를 둡니다: `kill`, `killall`, `pkill`, `rm`, `rmdir`, `dd`, `mkfs`, `shutdown`, `reboot`, `truncate`, `drop`, `/dev/` 대상 리다이렉션 등을 탐지.
  - `confirm_mode=all`이면 모든 실행 전 확인.
  - `confirm_mode=destructive`이면 위험 명령만 확인.
  - `confirm_mode=none`이면 바로 실행.
  - 복사 모드에서는 확인 프롬프트를 띄우지 않습니다.

## 공개 인터페이스 / 타입
- CLI 공개 인터페이스는 `qc`, `qc init`, `qc config show`, `--execute`, `--model`, `--host`, `--confirm`으로 고정합니다.
- 내부 백엔드 인터페이스는 `generate(task, context) -> ModelReply` 형태의 추상화로 두고, v1의 `ModelReply`는 `CommandReply | ClarificationReply` 두 타입만 허용합니다.
- 설정 스키마는 flat + ollama 필드 중심으로 시작합니다.
- v1에서 provider 선택지는 실사용상 `ollama`만 노출하되, 코드 구조는 이후 OpenAI-compatible provider를 추가할 수 있게 유지합니다.

## 테스트 계획
- 단위 테스트
  - 기본 모드가 `copy`인지.
  - `--execute`가 mode를 정확히 override하는지.
  - config/env/CLI 우선순위가 맞는지.
  - destructive matcher가 `rm -rf`, `kill`, `dd` 등을 잡는지.
  - structured output 파서가 `command`와 `clarification`을 올바르게 구분하는지.
  - `ollama list` 파싱이 `qwen3.5:9b` 우선 선택 규칙을 지키는지.
- 통합 테스트
  - mock Ollama 서버가 `command` JSON을 주면 명령이 출력되고 복사 모드가 성공하는지.
  - mock Ollama 서버가 `clarification` 후 `command`를 주면 재질문 후 완료되는지.
  - malformed JSON 응답 시 명확한 오류 메시지로 실패하는지.
  - 실행 모드에서 `confirm_mode`별 분기가 맞는지.
- 수동 스모크 테스트
  - macOS에서 `pbcopy`가 실제로 동작하는지.
  - `qc "1380 포트 쓰는 프로세스 찾아"`가 명령을 출력하고 클립보드에 들어가는지.
  - `qc --execute "pwd"`가 현재 셸 기준으로 실행되는지.

## 가정과 기본 선택
- v1은 `pls`의 “명령 실행을 관찰하며 반복하는 에이전트”가 아니라 “좋은 단일 명령을 제안하는 로컬 우선 도구”로 정의합니다.
- 실행 모드보다 복사 모드가 제품의 기본 정체성입니다.
- 설치된 로컬 모델 중 현재 환경에서는 `qwen3.5:9b`가 가장 적절한 기본값으로 간주합니다.
- 원격 provider는 이번 범위에 포함하지 않지만, provider 경계는 미리 분리합니다.
- Ollama structured output과 tool-calling 가능 여부는 공식 문서로 확인했습니다: [Structured Outputs](https://docs.ollama.com/capabilities/structured-outputs), [Tool calling](https://docs.ollama.com/capabilities/tool-calling), [Chat API](https://docs.ollama.com/api/chat)
